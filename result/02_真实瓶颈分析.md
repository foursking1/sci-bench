## 二、根因详细分析

### 根因1：实验范围严重缩水（91条规则，10篇论文）

**定义**：论文仅执行了作者复现代码中的部分功能，大量对比实验、消融实验、多模型/多数据集评估未运行，导致truth.md中对应规则无法验证。

这是**最大的失败来源**，占总失败的37%。其典型特征是：实验代码存在且可运行，但truth.md要求的多数实验路径在metrics.json中缺失（`expression evaluation failed` / `value not found`）。

## 四、具体改进建议

### 针对根因1：实验范围严重缩水

**核心问题**：truth.md覆盖论文的全部实验，但复现代码仅实现其中一部分。

1. **truth.md范围收敛**：对partial模式的论文，truth.md应仅覆盖实际已实现和已运行的实验。在创建truth.md之前，先扫描作者代码目录，确认哪些P-metrics有对应代码。

2. **规则创建前做能力评估**：为每条规则增加一个"可执行性"检查，如果作者代码中不存在对应的实验脚本或数据路径，该规则标记为"not_applicable"而非创建后永远失败。

3. **复现报告结构化分析**：在collect阶段解析reproduction_report.json或CHECKLIST.md中的"limitations"和"deviations"部分，自动将受限实验从truth.md中排除。

4. **优先级**：高。解决后可消除约91条（37%）失败规则。

### 针对根因2：数值系统性偏差

**核心问题**：实验跑了但数值对不上。

1. **区分"量级错误"和"精度偏差"**：
   - 量级错误（差>10倍）：标记为实验条件不一致，需要检查数据处理管线
   - 精度偏差（差<2倍但超tolerance）：可能是seed/参数差异，考虑放宽tolerance

2. **tolerance自适应**：对已知容易波动的指标（如训练loss、收敛指标），使用百分比tolerance（如10%）而非固定绝对值。

3. **检查数据提取准确性**：部分target_value从论文图表中OCR提取，本身可能有误差。对关键规则用原文数值表确认。

4. **统一尺度约定**：确认百分比指标是0-1还是0-100，确保truth.md和metrics.json使用相同尺度。

5. **优先级**：高。解决后可消除约40-50条（16-20%）失败规则。

### 针对根因3：关键数据集/外部依赖不可用

**核心问题**：基础设施约束导致实验无法完成。

1. **数据集预检查**：在truth.md创建前，检查所需数据集是否可下载。不可用的数据集对应的规则标记为"blocked"。

2. **替代方案注册**：对不可用的外部工具（Gurobi/Marabou），truth.md应允许定义替代验证方法，而非直接跳过。

3. **GPU预算标注**：对需要>8 GPU-hours的实验，truth.md中标注所需资源，允许在资源不足时降级验证。

4. **优先级**：中。需基础设施配合，短期内可通过规则排除缓解。

### 针对根因4：图表未生成

**核心问题**：90%的图表缺失是实验未执行的直接结果。

1. **随根因1/3解决后自动改善**。

2. 对于实验已执行但图表未生成的2%情况，检查绘图代码是否被正确调用。

3. **优先级**：低（已覆盖在根因1/3中）。

### 针对根因5：LLM评估不稳定

**核心问题**：LLM对科学图表评分偏低或拒绝评分。

1. **优化rubric**：当前rubric的3.0阈值可能过高。建议：
   - 将阈值降至2.5/5.0或2.0/3.0
   - 提供更具体的视觉元素描述而非主观质量判断
   - 对"LLM judgment failed"增加retry机制（换prompt重试）

2. **图像预处理**：确保导出图表有足够分辨率和清晰的轴标签。

3. **提供ground truth参考**：在LLM prompt中提供论文原图描述作为参考。

4. **优先级**：中。解决后可消除约11条（4.5%）失败规则。

### 针对根因6：truth.md规则不匹配

**核心问题**：规则生成逻辑对partial/exists规则处理有缺陷。

1. **exists规则的值匹配逻辑修复**：当前exists规则同时检查"文件存在"和"值匹配"，但"值匹配"在某些场景下无意义（如期望72.0但实际是布尔值True）。

2. **partial模式规则过滤**：对partial模式论文，不创建需要完整实验覆盖的规则。

3. **优先级**：低。仅影响10条规则，但修复成本低。

## 六、结论

**系统瓶颈的核心不是"证据采集不足"，而是"truth.md规则的期望覆盖了复现系统实际无法达成的实验"**。

246条失败规则中：
- **约50%（129条）** 是因为truth.md要求的实验在复现代码中未实现或未运行，属于**规则期望过高**。这些可以通过收窄truth.md范围、标记blocked实验为not_applicable来解决。
- **约31%（76条）** 是数值偏差，其中约一半是**严重量级偏差**（实验条件不一致），另一半是**中等偏差**（可调整tolerance或修复数据处理）。
- **约5%（11条）** 是LLM评估问题，可通过优化rubric解决。
- **约4%（10条）** 是规则生成逻辑缺陷，修复成本低。

**最高优先级的改进**：在truth.md创建阶段加入"复现能力评估"环节，自动识别哪些实验在当前环境下可执行、哪些需要标记为blocked/not_applicable。这将一次性消除约50%的失败规则。
